# 🎧 Model Card — Music Recommender with RAG

## 1. Model Name

**TuneTailor 2.0** — a content-based music recommender extended with a Retrieval-Augmented Generation (RAG) layer for natural-language explanations.

---

## 2. Intended Use

This system is designed to recommend a small list of songs (top 5) from a curated catalog based on a user's stated taste profile, and to **explain those recommendations in warm, human-sounding language** rather than just numeric scores. The explanations are generated by Claude (Anthropic's LLM) using rich descriptions retrieved from a knowledge base.

**Who it's for:** Educational and simulation purposes only — built as part of an applied AI course to demonstrate how content-based recommenders, retrieval systems, and large language models can be combined into a single trustworthy AI pipeline. It is **not** intended for real-world deployment with actual users.

---

## 3. How It Works (Short Explanation)

The system runs in two stages:

**Stage 1 — Scoring (the math):**
Every song in the catalog is compared against the user's preferences across 7 features (genre, mood, energy, danceability, valence, tempo, and acousticness). Each feature contributes a weighted score, and the songs are ranked from highest to lowest total score. The top 5 are selected.

**Stage 2 — Explanation (the prose):**
For each top-5 song, the system **retrieves** a hand-curated description from a knowledge base (`data/song_descriptions.csv`), **augments** an LLM prompt with the user's preferences, the song's metadata, the score breakdown, AND the description, then asks Claude to **generate** a 2-3 sentence natural-language explanation. This is the classic RAG pattern: retrieve, augment, generate.

The final output shows **both** the original numeric reasons (like "Genre match (+0.25)") **and** the AI-generated prose, so the user can always verify the AI's reasoning against the underlying math.

---

## 4. Data

**Catalog** — `data/songs.csv` contains 27 fictional songs across diverse genres, including pop, lofi, rock, ambient, jazz, synthwave, indie pop, electronic, reggae, country, classical, hip-hop, metal, blues, funk, R&B, folk, techno, gospel, ska, world, and new age. Each song has 10 attributes (id, title, artist, genre, mood, energy, tempo_bpm, valence, danceability, acousticness).

**Knowledge base** — `data/song_descriptions.csv` is new for this version. It contains 27 hand-curated descriptions (2-3 sentences each) matched to the songs by ID. The descriptions use a mix of styles: vibe-focused, story-focused, and recommendation-focused framing.

**Limits of the data:** The dataset is small (27 songs), and the descriptions reflect a single curator's perspective on each track's "vibe." Real-world recommenders would draw from millions of songs and incorporate listener behavior data, neither of which this system has access to.

---

## 5. Algorithm Summary

The scoring rules are weighted as follows:

- **Genre match:** +0.25 (highest weight — strong signal for music taste)
- **Mood match:** +0.20
- **Energy similarity:** up to +0.15 based on closeness to user's target
- **Danceability similarity:** up to +0.15
- **Valence similarity:** up to +0.15
- **Tempo similarity:** up to +0.05 (normalized across the catalog)
- **Acousticness preference match:** up to +0.05

Songs accumulate points across all 7 features and are ranked by total score. The RAG layer activates only on the top 5, calling Claude (`claude-haiku-4-5`) once per recommendation to generate the explanation.

---

## 6. Observed Behavior and Biases

Several biases and limitations were identified through testing:

**Genre dominance bias.** With genre weighted at +0.25 (the highest weight), songs in the user's favorite genre almost always rank near the top, even when other features are mediocre matches. This creates a "filter bubble" effect where users rarely get exposed to songs outside their stated genre preference.

**Dataset size bias.** With only 27 songs, certain niche genres (like "metal" or "gospel") have only one representative each. A user who prefers those genres will see the same song every time, with no variety.

**Curator bias in the knowledge base.** The 27 song descriptions were written from a single perspective. Concepts like "happy," "intense," or "moody" reflect one cultural and personal interpretation of those moods. A user from a different cultural background might find the AI explanations less resonant.

**LLM-specific risks.** The Claude API can occasionally generate explanations that emphasize features that weren't actually the strongest match factors. Because the user sees both the AI prose AND the numeric reasons, they can spot these mismatches — but a less attentive user might trust the prose without verifying.

**Adversarial behavior.** When tested with profiles that had nonexistent genres or conflicting preferences (e.g., high energy + sad mood), the system gracefully returned its best available matches rather than crashing, but the AI explanations sometimes felt forced as the LLM tried to justify weak matches.

---

## 7. Evaluation Process

The system was tested with **6 distinct user profiles**:

1. Indie pop / chill (standard)
2. Rock / intense (standard)
3. Pop / happy (standard, default test profile)
4. Adversarial: lofi / "any" mood / extremely high energy (conflicting preferences)
5. Adversarial: nonexistent genre / intense (testing fallback behavior)
6. Adversarial: nonexistent genre / "any" mood / neutral targets / acoustic preference (multiple edge cases)

For each profile, the top 5 recommendations were inspected for accuracy, reasoning quality, and robustness. Screenshots of the standard pop/happy profile and one adversarial profile are included in the README.

The system also has automated tests in `tests/test_recommender.py` covering the core scoring logic.

**One key surprise during testing:** When the AI explanations were enabled, profiles that previously felt mathematically valid sometimes produced explanations that revealed *why* those matches felt off intuitively. The prose layer surfaced limitations that the numeric output had hidden.

---

## 8. Intended Use and Non-Intended Use

**Designed for:** Demonstrating the RAG pattern, exploring how scoring engines and LLMs can work together, and showing how dual-output design (math + prose) makes AI systems more trustworthy.

**NOT designed for:** Real production music recommendation. Real-world deployment with paying users. Any decision-making context where someone might rely on the AI's explanation without verifying the underlying scoring data. Generating descriptions for songs that aren't in the curated knowledge base — the system would fall back to a generic message.

---

## 9. Ideas for Improvement

If I kept developing this, I'd:

1. **Add semantic retrieval.** Replace the simple ID-based lookup with a vector embedding search, so the system could match descriptions to user *intent* (e.g., "songs for a rainy night") rather than relying on exact song IDs. This would also unlock support for a much larger catalog.
2. **Build an evaluation harness.** Create an automated script that runs the system against many profiles and scores explanation quality (relevance, accuracy, tone) — moving beyond the current human-eyeball evaluation.
3. **Add diversity logic.** Implement a "diversity penalty" that prevents the same artist or genre from dominating the top 5, addressing the genre-bubble bias.

## 10. Personal Reflection

### What surprised me while testing my AI's reliability?

What surprised me most was how much the **AI explanation layer revealed about the scoring logic itself**. When I ran the system without RAG, the recommendations felt mathematically sound, but I couldn't always tell *why* a match worked or didn't. Once Claude was generating explanations grounded in the song descriptions, it became obvious when a song was technically a high-scoring match but didn't actually feel right for the user — because the AI would either struggle to justify it or produce prose that exposed the mismatch. The prose layer, in other words, became an unexpected debugging tool for the math layer.

### Limitations and biases in my system

The most significant limitation is the **small dataset** — 27 songs cannot meaningfully represent the diversity of real music taste, and certain genres have only one representative track. Beyond that, my **scoring weights reflect my personal assumptions** about what makes a good music match (I weighted genre highest), and the **knowledge base reflects my single-curator perspective**, which could feel less resonant to users from different cultural backgrounds. Finally, the system is fully **content-based** — it has no awareness of listener behavior, novelty, or surprise, which means it tends to reinforce existing taste rather than expand it.

### Could this AI be misused, and how would I prevent that?

In a real product, the warm AI-generated explanations could become a **manipulation surface**. If the prose were weighted too heavily in a user's perception, a recommender could nudge users toward sponsored content while making it sound like a perfect match. The dual-output design I built — showing both the score breakdown AND the prose — is one mitigation, because it gives users a way to verify the AI's claims against the underlying numbers. In a production version, I would also rate-limit how often the AI can call certain "persuasion patterns" (like emotional language or exclusivity claims), and I'd flag explanations that diverge significantly from the structured score data.

### Describing my AI collaboration

I used Claude (in the Claude.ai chat interface) extensively throughout this project, and the experience taught me that AI collaboration is most useful when I treated it as a thinking partner rather than a code generator.

**One particularly helpful suggestion** came when I was deciding how to display the RAG output. I was leaning toward replacing the original "Reasons" line entirely with the AI explanation — making it look slicker. Claude suggested instead keeping **both** outputs visible: the math-based reasons AND the AI prose underneath. The reasoning was that this would make my system more transparent and trustworthy, because users could always verify the AI's claims against the underlying score data. I hadn't framed my project as a "responsible AI design" exercise before that moment, but once I made that choice, the responsibility framing tied the whole project together — it became the central thread of my model card and presentation.

**One flawed suggestion** came during integration. Claude told me to use `from recommender import load_songs, recommend_songs` in `main.py`, but when I ran the script with `python -m src.main`, I got a `ModuleNotFoundError: No module named 'recommender'`. The fix was to use `from src.recommender import...` to match the package structure. The AI's initial suggestion had ignored the difference between running a script directly versus running it as a module — a subtle distinction that mattered. This reminded me that I always need to verify AI-generated code by actually running it, because plausible-looking code can still be subtly wrong.

### What would I try next?

If I extended this project further, I'd focus on **adding semantic retrieval** — moving from "look up description by song ID" to "find descriptions that semantically match the user's request." That would unlock natural-language queries like *"recommend me something for a rainy night"* and demonstrate the full power of RAG beyond the simple lookup pattern I built here. I'd also want to build a proper evaluation harness so I could measure improvements rigorously, instead of relying on intuition.
